# Архитектура и MLOps

## Выбор архитектуры

Для учебного проекта выбран монолитный веб-сервис: Flask-приложение загружает сохранённые модели, валидирует JSON-запрос, выполняет инференс и возвращает прогноз.

Такой подход оправдан, потому что:

- модель небольшая и не требует отдельного model-serving кластера;
- нет тяжёлой асинхронной обработки в основном пользовательском сценарии;
- проще воспроизвести проект локально, в Docker и при проверке;
- меньше инфраструктурных зависимостей.

Микросервисный подход стал бы полезен при нескольких продуктовых командах, большом количестве моделей, независимых релизах feature-store/model-serving/API или при необходимости масштабировать обучение, инференс и мониторинг отдельно.

## Поток запроса

```text
Client
  -> Flask /predict
  -> JSON validation
  -> A/B routing: v1 control or v2 test
  -> sklearn Pipeline: feature engineering -> preprocessing -> classifier
  -> JSON response
  -> JSONL request log
```

## Логирование и мониторинг

Сервис пишет одну JSON-строку на каждый успешный прогноз в `logs/api_requests.jsonl`.

Поля события:

- `ts`: UTC timestamp;
- `client_id`: идентификатор клиента, если передан;
- `model_version`: выбранная версия модели;
- `ab_group`: `control` или `test`;
- `prediction`: бинарный прогноз;
- `default_probability`: вероятность дефолта;
- `risk_level`: категория риска на основе вероятности дефолта.

В production эти логи можно отправлять в stdout контейнера и собирать через Filebeat/Fluent Bit в ELK/OpenSearch. На основе логов стоит строить мониторинг:

- количества запросов и ошибок;
- распределения вероятностей дефолта;
- доли клиентов, направленных в control/test;
- задержки API;
- drift признаков относительно обучающей выборки;
- постфактум качества модели, когда становятся доступны реальные дефолты.

## RabbitMQ и очереди

RabbitMQ в текущей реализации не нужен, но при масштабировании он может пригодиться для:

- асинхронной обработки batch-заявок;
- фоновой записи аудита и обогащённых логов;
- отправки событий в систему мониторинга риска;
- повторной обработки запросов при временных ошибках downstream-сервисов.

Пример сценария: API быстро принимает batch-файл, кладёт задачу в очередь `credit_scoring_batches`, worker читает задачу, делает предсказания и сохраняет результат в хранилище. Пользователь получает `job_id`, а не ждёт длинный HTTP-запрос.

В текущей учебной версии добавлен синхронный endpoint `POST /batch_predict`. Он подходит для демонстрации batch-инференса на небольшом количестве записей. При росте объёмов его лучше заменить на асинхронную схему с очередью.

## DVC

DVC полезен для версионирования данных и связки `код -> данные -> модель`. В этом проекте DVC мог бы хранить:

- исходный UCI CSV;
- train/test split;
- обученные `.joblib` модели;
- метрики экспериментов.

Это позволило бы воспроизвести конкретный релиз модели и понять, на каких данных он был обучен.

## MLflow

MLflow полезен для трекинга экспериментов:

- параметры модели;
- метрики `precision`, `recall`, `f1`, `roc_auc`;
- артефакты моделей;
- сравнение v1/v2;
- регистрация production/staging версий.

В промышленном варианте MLflow Model Registry можно использовать как источник моделей для сервиса вместо локальной папки `models/`.

## ONNX-ML

Для оптимизации инференса sklearn pipeline можно конвертировать в ONNX-ML с помощью `skl2onnx`.

Общий план:

```python
from skl2onnx import convert_sklearn
from skl2onnx.common.data_types import FloatTensorType

initial_type = [("float_input", FloatTensorType([None, n_features]))]
onnx_model = convert_sklearn(pipeline, initial_types=initial_type)
with open("model.onnx", "wb") as f:
    f.write(onnx_model.SerializeToString())
```

В реальном проекте нужно отдельно проверить корректность предобработки категориальных признаков, совпадение вероятностей sklearn и ONNX Runtime, а также latency на целевой инфраструктуре.

## uWSGI/Gunicorn и NGINX

Flask development server не предназначен для production. В Dockerfile используется Gunicorn: он управляет worker-процессами и устойчивее обслуживает HTTP-запросы.

NGINX обычно ставят перед WSGI-сервером для:

- TLS termination;
- reverse proxy;
- лимитов размера запроса;
- rate limiting;
- сжатия ответов;
- отдачи статических файлов;
- балансировки между несколькими инстансами API.

uWSGI решает похожую задачу запуска Python WSGI-приложений, но для этого проекта Gunicorn проще и достаточен.

## Бизнес-метрики

Технических метрик недостаточно: банк интересуют деньги и риск.

1. Ожидаемое снижение финансовых потерь.
   Можно рассчитать как сумму ожидаемых потерь по клиентам, которым модель отказала или отправила на ручную проверку:
   `PD * EAD * LGD`, где `PD` берётся из `default_probability`, `EAD` можно приближать кредитным лимитом, `LGD` задаётся бизнесом.

2. Доля одобренных заявок при фиксированном уровне риска.
   Для выбранного порога риска сравнивается, сколько клиентов можно одобрить при ограничении, например, `bad rate <= 5%`.

3. Стоимость ручной проверки.
   Если модель отправляет часть клиентов на manual review, нужно учитывать экономию от предотвращённых дефолтов и операционные расходы на проверку.
