# План A/B-теста

## Цель

Сравнить текущую модель `v1` с новой моделью `v2` и проверить, улучшает ли `v2` качество выявления дефолтов без недопустимого роста ложных срабатываний.

## Группы

- Control: `v1`, текущая модель.
- Test: `v2`, новая модель.

Если `model_version` не передан в `/predict`, сервис выполняет стабильное распределение 50/50 по `ID` клиента или по хешу тела запроса. Это важно: один и тот же клиент должен попадать в одну и ту же группу.

## Длительность

Рекомендуемая длительность: 4-8 недель или до накопления заранее рассчитанного минимального размера выборки. Для кредитного дефолта бизнес-лейбл появляется не мгновенно, поэтому финальный анализ качества нужно проводить после окна наблюдения за фактом дефолта.

## Технические метрики

Основная метрика:

- F1-score для класса дефолта. Она балансирует precision и recall и подходит при дисбалансе классов.

Дополнительные метрики:

- Recall для класса дефолта: важно не пропускать рискованных клиентов.
- Precision для класса дефолта: важно не блокировать слишком много хороших клиентов.
- ROC-AUC: полезен для оценки ранжирования риска независимо от фиксированного порога.

## Бизнес-метрики

1. Expected loss reduction:
   `sum(default_probability * exposure_at_default * loss_given_default)` по клиентам, которым отказали или которых отправили на ручную проверку.

2. Approval rate under risk constraint:
   доля одобренных заявок при фиксированном допустимом уровне дефолта.

3. Manual review load:
   доля клиентов, попавших в серую зону риска и требующих ручной проверки.

## Статистический анализ

Для долей ошибок, precision и recall можно использовать z-test для двух пропорций и доверительные интервалы Уилсона или bootstrap.

Для F1-score, так как это нелинейная метрика, предпочтителен bootstrap:

1. Сэмплировать клиентов с возвращением отдельно или совместно по группам.
2. На каждой итерации считать F1 для `v1` и `v2`.
3. Получить распределение `delta = F1_v2 - F1_v1`.
4. Построить 95% доверительный интервал по перцентилям.

## Критерий успеха

`v2` считается успешной, если выполняются условия:

- 95% доверительный интервал для `F1_v2 - F1_v1` выше 0 или нижняя граница не ниже заранее согласованной non-inferiority границы;
- recall класса дефолта не ухудшается больше допустимого порога;
- precision не падает настолько, чтобы резко увеличить долю ложных отказов;
- бизнес-метрика expected loss улучшается или approval rate растёт при том же уровне риска.

## Связь с архитектурой

Текущий API уже поддерживает A/B-тестирование:

- явное тестирование через `model_version`;
- автоматическое стабильное 50/50 распределение;
- логирование `model_version` и `ab_group`.

В production дополнительно нужны:

- централизованное хранилище фактических исходов дефолта;
- join логов предсказаний с бизнес-результатами;
- защита от sample ratio mismatch;
- алерты при деградации API или перекосе трафика между группами.
